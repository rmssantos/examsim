# Per-exam SEO landing pages — Design

Date: 2026-06-03
Status: Approved (design), pending spec review
Owner: Ruben

## Problem

Today `examplar.app` exposes a single indexable page (`index.html`). The exam
library (SC-900, AZ-900, AB-730, ...) is rendered by JavaScript, so a crawler
sees an almost empty page. People search for queries like "SC-900 practice
exam" or "free AZ-900 practice questions"; we have nothing static to rank for
them.

## Goal

Ship one static, pre-rendered landing page per exam, rich with unique content
drawn from each exam's `metadata.json`, so each page can rank for that exam's
keywords and funnel users into the existing app.

## Non-negotiable constraint: do not break anything that works today

This feature is **additive**. It must not change the behaviour of the existing
SPA, routing, service worker, or any current URL.

- **New files only** (no edits to app JS/CSS logic):
  - `tools/generate-exam-pages.py`
  - `tools/exam-page-template.html` (and a small hub template)
  - `assets/css/exam-landing.css`
  - `exams/index.html` and `exams/<id>/index.html` (generated, committed)
- **Edits to existing files are minimal and additive:**
  - `sitemap.xml` — regenerated by the tool (already a standalone file).
  - `index.html` — add ONE crawlable link to `/exams/` (placed in the existing
    footer). No change to the SPA screens, scripts, or card logic.
  - `.github/workflows/deploy-pages.yml` — add a build step that runs the
    generator before the artifact upload.
- **No changes** to `exam.html`, `editor.html`, the router, `script-multi-exam.js`,
  or any existing query-param behaviour. Landing pages link OUT to the existing,
  already-working URL `exam.html?exam=<id>`; they never modify it.
- **Service worker:** the new `/exams/...` pages are content pages. They are NOT
  added to the precache `CORE_ASSETS` list, so the install/activate flow is
  unchanged. We must verify the existing `fetch` handler falls through to the
  network for unknown paths (expected: runtime cache / network passthrough), so
  the new pages load normally and the SW keeps working as-is.
- **Verification before done:** existing flows still work (home loads, exam
  launches, editor opens, privacy page opens, PWA installs/works offline as
  before). New pages return 200 at their clean URLs and link correctly into the
  app.

## URL structure

Folder + `index.html` so the clean URL works WITHOUT JavaScript (this is what
makes it crawlable, unlike the SW-dependent clean routes used by the app):

- `https://examplar.app/exams/` — hub page ("All practice exams")
- `https://examplar.app/exams/sc900/` — one per exam

## Generation (build-time, Python, zero dependencies)

`tools/generate-exam-pages.py`, following the style of the existing
`tools/inject-analytics-secret.py`:

1. Read `user-content/exams/index.json` (the ordered list of exam ids).
2. For each id, read `user-content/exams/<id>/metadata.json`.
3. Render `tools/exam-page-template.html` with that metadata to
   `exams/<id>/index.html`.
4. Render the hub `exams/index.html` linking to every exam.
5. Rewrite `sitemap.xml` to include `/`, `/exams/`, every `/exams/<id>/`, and
   `/privacy-and-storage.html`. The generator becomes the single source of truth
   for the sitemap so it never drifts.

Missing/empty exam folders (e.g. an `ai901` dir with no `metadata.json`) are
skipped with a warning, never crash the build.

### Output is committed to git

Generated `exams/**` and the updated `sitemap.xml` are committed (they contain no
secrets, unlike the analytics string). Reasons: local preview via `server.py`
works immediately, the homepage `/exams/` link is never a 404 in dev, and PR
reviewers see the real output. To prevent drift, CI re-runs the generator and
fails if the working tree changes (a "generated files are stale" check), and the
deploy job regenerates before upload anyway.

## Page content (rich, unique per exam, from metadata)

Each `exams/<id>/index.html` contains:

- `<head>`: keyword title (e.g. `SC-900 Practice Exam (Free, No Sign-up) | Examplar`),
  generated meta description, self canonical, Open Graph + Twitter tags,
  `theme-color`, favicon/manifest links (mirroring the existing pages).
- H1 `<exam code> Practice Exam` + subhead with `fullName`.
- Intro paragraph: free, private, fully offline, original questions (brand voice).
- Primary CTA button → `exam.html?exam=<id>` ("Start practicing free").
- Exam facts: vendor, certification code, level/badge, duration, questions per
  attempt (`questionCount`), question bank size (`totalQuestions`), pass score.
- Objective domains: code, name, weight range % (rich, exam-specific text).
- Modules covered (from `modules`).
- Official resources (from `resources`, external links).
- FAQ (4-5 Q&A; generic + exam-specific) used for the FAQPage rich result.
- Cross-links to the other exams (internal linking).
- Existing site header (logo, theme toggle) and footer for consistency.

### Structured data (JSON-LD) per page

- `Course` describing the free practice exam (with `provider` Examplar and the
  fields Google's Course rich result needs).
- `BreadcrumbList`: Home > Exams > <code>.
- `FAQPage` matching the on-page FAQ.

## Internal linking / crawlability

Crawl path guaranteed without relying on JS:
`index.html` (footer link) -> `/exams/` hub -> each `/exams/<id>/` -> CTA into
`exam.html?exam=<id>` and cross-links between exam pages.

Upgrading the homepage exam *cards* to anchor directly at the landing pages is a
possible phase 2; it is intentionally OUT of scope here to avoid touching app JS.

## Styling

Reuse the existing stylesheets for header/theme/footer; add a small
`assets/css/exam-landing.css` for the landing layout only. Visual language
matches the current site.

## Copy / brand voice constraints

- No em-dashes in any generated copy.
- Never use "dump" terminology in visible copy; say "questions" / "practice
  pack". (The internal data filename `dump.json` is unrelated and untouched.)

## Build integration

Add to `.github/workflows/deploy-pages.yml`, before "Upload artifact":

```yaml
- name: Generate per-exam SEO pages
  run: python tools/generate-exam-pages.py
```

(Optionally a separate CI freshness check in `validate.yml`.)

## Out of scope (YAGNI)

- Per-exam custom hand-written copy overrides (the rich metadata is enough).
- Homepage card anchor rewrite (phase 2).
- Performance work (Font Awesome subset, CSS minify) — tracked separately.
- Search Console / Bing submission — manual, outside the code.

## Success criteria

- Each exam has a 200-OK page at `examplar.app/exams/<id>/` with unique,
  metadata-driven content and valid JSON-LD (Course + BreadcrumbList + FAQPage).
- `sitemap.xml` and the `/exams/` hub list every exam.
- All existing flows (home, exam, editor, privacy, PWA offline) behave exactly
  as before. Nothing currently working is broken.
